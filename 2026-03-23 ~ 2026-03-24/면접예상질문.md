> 
> 브라우저가 URL에 적힌 값을 파싱해서 HTTP Request Message를 만들고 Os에 전송을 요청한다 이때 Domain으로 요청을 보낼 수 없기 때문에 DNS Lookup을 수행합니다
>
> DNS Lookup과정은 크롬의 경우라면 브라우저 -> hosts 파일 -> DNS cache의 순서로 도메인에 매칭되는 ip를 찾습니다 일반적으로 설명하는 DNS Lookup은 루트 도메인서버에서부터 서브 도메인 서버순으로 찾게 됩니다
>
> 이 요청은 프로토콜 스택이라는 OS에 저장되어 있는 네트워크 제어용 소프트웨어에 의해 패킷에 담기고 패킷에 제어정보를 덧붙여 Lan 어댑터에 전송하고 lan 어댑터는 이를 전기신호로 변화시켜 송출합니다
>
> 패킷은 스위칭 허브등을 경유하여 인터넷 접속용으로 라우터에서 ISP로 전달되어 인터넷으로 이동합니다<br>액세스 회선에 의해 통신사용 라우터로 운반되고 인터넷의 핵심부로 전달됩니다 고속 라우터들 사이로 목적지 까지 패킷이 흘러들어가게 됩니다
>
> 핵심부를 통과한 패킷은 목적이인 LAN에 도착하게 되고 방화벽이 패킷을 검사하고 캐시 서버로 보내어 웹 서버에 갈 필요가 있는지 검사합니다
>
> 웹 서버에 도착한 패킷은 프로토콜 스택이 패킷을 추출하여 메시지를 복원하고 웹 서버 애플리케이션에 넘깁니다 애플리케이션은 요청에 대한 응답 데이터를 작성하여 클라이언트로 회송하고 이는 전달된 방식 그대로 전송됩니다

> **TCP와 UDP의 차이점에 대해 설명하시오**<br>
> TCP는 연결 지향적 프로토콜이고 UDP는 데이터를 데이터그램단위로 전송하는 프로토콜입니다
>
> TCP는 가상 회선을 만들어 신뢰성을 보장하도록 하는 프로토콜로 따로 신뢰성을 보장하도록 하긴 위한 절차 때문에 절차가 없는 UDP보다 속도가 느린편입니다
>
> 그래서 TCP는 파일전송과 같은 신뢰성이 중요한 곳에 사용되고 UDP는 스트리밍, RTP와 같은 연속성이 중요한 곳에 사용이 됩니다
>
> 하지만 UDP가 신뢰성을 보장하지 않을 뿐이지 사용자가 직접 신뢰성을 보장하도록 할 수 있습니다 그래서 HTTP/3은 QUIC이라는 프로토콜 기반으로 하는데 QUIC는 UDP를 기반으로 합니다 즉 UDP 자체는 신뢰성을 보장하진 않지만 추가적인 정의를 통해 신뢰성을 보장받을 수 있습니다

> **TCP3,4 way handshake에 대해 설명하시오**<br>
> TCP 3way handshake는 가상 회선을 수립하는 단계입니다 클라이언트는 서버에 요청을 전송할 수 있는지 서버는 클라이언트에게 응답을 할 수 있는지 확인하는 과정입니다 SYN ACK 패킷을 주고받으며 임의의 난수로 SUN 플래그를 전송하고 ACK 플래그에는 1을 더한값을 전송합니다 정확한 순서로는 SYN(n) -> ACK(n+1), SYN(m) -> ACK(m+1) 순으로 일어납니다
> 왜 임의의 난수로 설정을 하냐면 기존 요청과 구분을 하기 위해서입니다
>
> TCP 4way handshake는 TCP연결을 해제하는 단계로 클라이언트는 서버에게 연결 해제를 통보하고 서버가 이를 확인하고 클라이언트에게 이를 받았음을 전송해주고 최종적으로 연결이 해제됩니다 단 서버에서 소켓이 닫혔다고 통지해도 클라이언트 측에서는 일정시간 대기를 하는데 이는 패킷이 나중에 도착 할 수 도 있기 때문이다

> **HTTP와 HTTPS의 차이점을 설명하시오**<br>
> HTTP는 따로 암호화 과정을 걸치지 않기 때문에 중간에 패킷을 가로 챌 수 있기 때문에 이를 해결하기 위해 나온 것이 HTTPS이다 HTTPS는 암호화 과정을 거쳐 패킷을 보내기 때문에 더욱 안전합니다

> **GET와 POST의 차이점에 대해 설명하시오**
> GET요청은 서버에 존재하는 정보를 요청하는 것 입니다 이때 반환되는 정보는 정보 자체가 아니라 정보의 표현입니다<br>
> POST요청은 서버에 정보를 생성하는 것을 요청합니다 예전 HTTP 통신은 POST 요청으로 데이터 삭제, 수정도 form요청으로 같이 수행합니다. POST 요청은 서버의 상태를 변경시키키 때문에 멱등성이 유지되지 않습니다

> **HTTP 메서드와 이것이 하는 역활에 대해 설명해보세요**<br>
> - GET요청은 서버에 존재하는 정보를 요청하는 것 입니다 CRUD에서는 R 부분 입니다
> - POST요청은 서버에 데이터를 생성하는 것을 요청하는 것 입니다 CRUD에서는 C 부분입니다
> - PUT요청은 서버에 존재하는 데이터를 수정하거나 없다면 생성하는 것 입니다 CRUD에서는 C,U입니다
> - DELETE요청은 서버에 데이터를 삭제하는 것을 요청하는 것 입니다 CRUD에서는 D 입니다
> - PATCH요청은 서버에 존재하는 데이터를 일부 수정하는 것을 요청하는 것 입니다 CRUD에서는 U 부분입니다

> **RESTful이란 무엇이며, 이에 대해서는 아는대로 설명하시오**<br>
> RESTful 이란 REST의 원리를 따르는 시스템을 의미합니다 하지만 REST를 사용 하였다고 모두가 RESTful한 것은 아닙니다 REST API의 설계 규칙을 올바르게 지킨 시스템을 RESTful하다 말할 수 있습니다 모든 CRUD를 POST로 구현한 REST API를 RESTful하다라고 할 수 없습니다

> **CORS란 무엇이며 이것에 대해 설명하시오**<br>
> CORS는 웹개발 하면서 가장 흔하게 만날 수 있는 문제 입니다 프론트 개발할 때 로컬에 API요청을 보낼때 나타납니다
> 
>  서로 다른 도메인 간에 자원을 공유하는 것을 의미합니다 대부분의 브라우저에서는 이를 기본적으로 차단하며 서버측에서 해더를 통해서 사용가능한 자원을 알려줍니다
>
> prefilght request는 실제 요청을 보내도 안전한지 판단하기 위해 사전에 보내는 요청입니다 OPTIONS 메서드로 요청하며 CORS를 허용하는지 확인합니다 CORS가 허용된 웹서버라면 사용 가능한 리소스를 헤더에 담아 응답합니다

> **OSI7계층과 그 존재 이유와 TCP/IP 4계층에 대해 설명하시오**<br>
> OSI7계층은 네트워크 구성 요소를 7개의 계층들로 나누어 표준화 시킨것 입니다 이렇게 표준화 하는 것의 장점은 통신이 일어나느 과정을 단계별로 파악할 수 있어 문제 해결에 용이합니다
>
> 실제로 우리가 대부분 사용하는 네트워크는 TCP/IP 4계층 입니다 통신에 실제로 사용되는 계층이고 1,2 계층이 1계층, 5,6,7계층이 4계층으로 운영됩니다

> **웹 서버 소프트웨어(Apache, Nginx)는 OSI 7계층 중 어디서 작동하는지 설명해보세요**<br>
> Apache와 NGINX는 HTTP 웹 서버로 이들이 작동하는 OSI 7 Layer 중 7계층인 애플리케이션 Layer 에 해당하는 프로토콜 입니다 HTTP 프로토콜은 TCP/IP 프로토콜을 통해 동작합니다 TCP/IP 프로토콜은 OSI 7 Layer 중 4계층인 Transport Layer 동작합니다 따라서 웹 서버 소프트웨어는 4계층의 TCP/IP와 7계층의 HTTP 프로토콜에서 작동합니다

> **웹 서버 소프트웨어(Apache, Nginx)의 서버 간 라우팅 기능은 OSI 7계층 중 어디서 작동하는지 설명해보세요**<br>
> 두가지가 있습니다 Layer 4 그리고 Layer 7입니다 L4에서 TCP/UDP 포트 정보를 토대로 라우팅 기능이 제공되고 L7에서는 TCP/UDP 뿐만 아니라 HTTP의 URI등을 토대로 라우팅이 기능이 제공 됩니다 L4에서는 라우팅 기능을 사용한 예시로는 NGINX 의 경우 여러 포트들을 하나의 upstream블록으로 묶어 로드 밸런싱 즉 특정 경로로 전달되는 요청을 각 포트별로 분산하여 전달 할 수 있도록 설정 해 줄 수 있습니다 L7에서 라우팅 기능을 사용한 예시로는 Apache, Nginx 각각에서 서브 도메인에 대해 라우팅 설정을 해둘 수 있습니다 브라우저에서 /test와 같은 서브 도메인으로 HTTP 프로토콜을 통한 요청을 보낸다면 웹 서버내 Config 파일에 설정된 경로의 정보를 토대로 요청에 대한 라우팅을 제공하여 스태틱 파일을 전달하거나 API 서버에 대해 리버스 프록시 역활을 해줄 수 있습니다
